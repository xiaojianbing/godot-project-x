# Feature Spec: Character 统一角色系统

## Feature 简述
一套**统一的角色数据与行为管理系统**，由三块基础能力共同构成：
- **CharacterAttributes（属性层）**：为所有角色提供基础属性模板、修正器和最终属性查询。
- **CharacterStats（资源层）**：为所有"可交互实体"提供当前 HP、当前能量等运行时资源值管理。
- **CharacterContext（行为层）**：为所有"有行为的实体"提供状态标志、动画桥接和组件引用枢纽。

覆盖：Player、NPC、Enemy、Boss、可破坏道具（箱子、门、可击碎墙壁等）。

## 玩家体验目标
- 玩家在不攻击时，仅凭跑、跳、冲刺、蹬墙等移动就能感到角色“黏手、顺手、可信”。
- 玩家攻击任何"可交互物体"（敌人、木箱、岩壁）时，获得**一致的打击反馈**——掉血、碎裂、闪烁。
- 不同对象被击中后有**符合直觉的差异化反应**：敌人硬直后退，木箱直接碎裂，Boss 不动如山但血条减少。
- 能量槽在战斗中**自然积攒**，玩家永远能看到自己距离大招还差多少。
- 大招霸体期间被攻击：正常扣血，但**不被打断**。

## 卖点对齐

- `CharacterStats` 与 `CharacterContext` 不只是底层基础设施，还必须服务项目的两大卖点：`丝滑操控` 与 `华丽战斗`。
- 任何数值、状态、受击、动作限制设计，都不能破坏输入响应、动作连续性和战斗观赏性。
- 当“规则严谨”与“操作爽感”冲突时，应优先寻找同时保留系统清晰度与主观手感的方案。

| 游戏           | 参考机制                                           | 对我们的启示                                  |
| -------------- | -------------------------------------------------- | --------------------------------------------- |
| 《空洞骑士》   | 敌人/可破坏物/玩家共用 HP 系统；灵魂值通过攻击积攒 | 统一的 Damageable 接口，攻击即积攒资源        |

| 《失落的皇冠》 | 能量通过击杀/处决/弹反积攒，速率不同               | 能量获取与战斗行为挂钩                        |
| 《仁王》       | 气力槽（Ki）影响攻防节奏                           | Buff/Debuff 对属性的动态修改                  |

## 优先级
**P0 基础设施** — 横跨 P0-A 至 P0-D 的底层数据+行为层，在 P0-A 阶段预埋。

## 范围定义

### In Scope

- 定义统一角色系统的产品目标、核心概念和实体覆盖边界。
- 定义 `CharacterStats` 作为统一数值层的职责范围。
- 定义 `CharacterContext` 作为行为层运行时枢纽的职责边界。
- 明确 Player、Enemy、Boss、NPC、可破坏道具在统一系统中的支持差异。
- 明确 HP、能量、Buff、状态标志、动画桥接等核心能力在各阶段的预埋目标。
- 明确 `EnemyCharacter` 这类真正角色实体默认应使用 `CharacterBody2D`、重力与统一角色运行时，而不是停留在纯 `Node2D` 靶子模型。
- 明确玩家视角验收标准与 P0-A ~ P0-D 的阶段完成定义。

### Out of Scope

- 具体类拆分、接口命名、模块通信方式与代码级 API 设计。
- HFSM 的具体状态树实现细节。
- Buff 数值公式、具体平衡参数与全部效果池。
- 完整战斗系统、连招系统、Boss 多阶段行为细节。
- 具体重生规则实现；玩家重生结果应遵循独立的 Respawn 规格。
- 关卡、房间、存档点、Boss 房编辑工具。

## 支持的实体类型

| 实体       | Stats (HP) | Stats (能量) | Stats (Buff) | Context (状态机) | Context (动画) | 示例             |
| ---------- | :--------: | :----------: | :----------: | :--------------: | :------------: | ---------------- |
| Player     |     ✅      |      ✅       |      ✅       |    ✅ 完整HFSM    |       ✅        | 主角             |
| Enemy      |     ✅      |      ❌       |      ✅       |    ✅ AI状态机    |       ✅        | 杂兵、精英怪     |
| Boss       |     ✅      |      ❌       |      ✅       |   ✅ 多阶段FSM    |       ✅        | Boss 战          |
| NPC        |   可选支持   |      ❌       |      ❌       |    ✅ 简化FSM     |       ✅        | 友方 NPC         |
| 可破坏道具 |     ✅      |      ❌       |      ❌       |        ❌         |       ❌        | 木箱、碎墙、灯笼 |

> 说明：`NPC` 是否拥有 HP 取决于具体交互类型，不要求所有 NPC 默认接入完整受伤/死亡规则。

> 补充：`Enemy` 一旦承担真实战斗与空间交互职责，就应接入 `CharacterBody2D`、重力、落地和基础运动约束；只有纯验证靶子才允许维持轻量非角色模型。

## 核心概念

### 三层架构总览

```
┌─────────────────────────────────────┐
│         StateMachine (HFSM)         │  ← 决定"该做什么"（各状态节点）
│   Idle / Run / Jump / Attack / ...  │
├─────────────────────────────────────┤
│        CharacterContext             │  ← "我现在是什么状态，能做什么"
│   状态标志 + 组件引用 + 动画桥接     │
├─────────────────────────────────────┤
│        CharacterStats               │  ← "我有多少数值资源"
│   HP / Energy / Buffs / Modifiers   │
└─────────────────────────────────────┘
```

- **StateMachine** 读 Context 判断能否切换，读 Stats / Attributes 判断资源与属性是否满足条件
- **Context** 是中间枢纽，持有对 Stats、Input、Physics 的引用
- **Attributes** 保存基础属性真相与最终属性结果
- **Stats** 保存当前资源值，不关心行为逻辑

---

### A. CharacterStats 模块（数值层）

#### A1. IDamageable 接口
所有可受击对象的统一入口。攻击系统只需调用 `IDamageable.TakeDamage(HitData)`，不关心对面是 Player 还是木箱。

#### A2. 属性值 (StatValue)
每个属性由 `BaseStat`（基础值）+ `Modifier`（修改器列表）构成，避免硬编码。Buff/装备/技能通过增减 Modifier 动态影响最终值。

#### A3. 能量槽 (Energy Gauge)
- 仅 Player 拥有
- 积攒来源：击杀(少) / 处决(中) / 弹反(多)（已确认，参考《失落的皇冠》）
- 消耗用途：P0-D 大招释放

> 产品口径：首版不定义“普通命中即获得能量”，避免与击杀 / 处决 / 弹反这套来源规则冲突。

#### A4. Buff 系统
- 临时性属性修改器，带持续时间
- 可叠加或互斥（按 BuffType 配置）
- 示例：攻击力+20% (5秒)、中毒持续掉血、移速降低

---

### B. CharacterContext 模块（行为层）

#### B1. 状态标志 (State Flags)
运行时标志，由当前状态机节点设置，供其他系统查询：

| 标志            | 类型 | 用途                    | 设置者                   |
| --------------- | ---- | ----------------------- | ------------------------ |
| IsGrounded      | bool | 地面判定                | 物理检测                 |
| IsInvincible    | bool | 无敌帧（Dash / 受击后） | Dash状态、HitStun状态    |
| HasSuperArmor   | bool | 霸体（大招期间）        | UltimateState            |
| CanAct          | bool | 能否接受输入            | HitStun/Death 设为 false |
| FacingDirection | int  | 朝向 (1/-1)             | 移动状态                 |

#### B1.1 手感保障

- 行为层必须支持 `Coyote Time`、`Jump Buffer`、输入缓存、动作优先级等手感保障机制。
- 角色在移动、受击、攻击、冲刺之间切换时，状态流转应优先保证响应清晰与结果可预测。
- 任何硬控、受击、霸体、连招限制都应说明它如何影响玩家的输入感知。

#### B2. 组件引用枢纽
Context 持有所有核心组件引用，状态机节点通过 Context 访问，无需各自 GetComponent：
- `CharacterBody2D` / 碰撞节点
- `CharacterStats`
- `PlayerInputHandler`（仅 Player）
- `Animator`
- `StateMachine`

#### B2.1 职责边界

`CharacterContext` 在产品层被定义为**行为层运行时枢纽**，用于承接“当前状态是什么、当前能否行动、其他系统如何访问角色运行时信息”这类需求。

它应负责：
- 暴露运行时状态标志
- 汇总行为层常用组件引用
- 作为状态机、动画、输入、物理之间的访问入口
- 对外提供统一的角色运行时查询点

它不应负责：
- 数值计算与 Buff 结算
- 完整受击编排和伤害公式
- 重生来源判定与重生点选择
- 房间、关卡、checkpoint、Boss 规则决策
- 演变成跨系统“上帝对象”

> 是否需要将 `CharacterContext` 进一步拆分为多个运行时协作对象，属于 `@Architect` 的设计任务，不在本 Feature Spec 中预先写死。

#### B3. 动画桥接
状态切换时由 Context 统一通知 Animator，避免各状态直接操作动画：
- `Context.PlayAnimation(string animName)`
- `Context.SetAnimParam(string param, value)`

#### B4. 受击响应协调

从产品层看，统一角色系统需要支持“受击后既发生数值变化，也可能触发行为变化”的结果。

产品要求是：
- 角色受击后应先产生正确的数值结果，例如扣血。
- 若角色当前不具备霸体或其他免打断条件，应能进入受击响应。
- 若角色具备霸体等高优先级保护，允许扣血但不进入常规受击硬直。

> 具体由 `CharacterContext`、专用受击协调器，还是其他运行时模块承担这段流程，交由 `@Architect` 设计。

#### B5. 连招表达支持

从产品层看，行为层需要为高观赏性的动作战斗预留稳定支点，包括但不限于：

- 输入缓存与队列执行
- 连招派生标志
- 取消窗口标志
- 挑空、击退、滞空和追击相关状态
- 霸体、弹反、处决等高优先级战斗结果

> `CharacterContext` 不必独自实现完整连招系统，但必须为其提供稳定、可查询、可扩展的运行时状态基础。

## 玩家视角验收标准

### 通用验收

1. 玩家攻击不同目标时，会得到一致但符合对象差异的反馈，不会觉得系统对敌人、箱子、Boss 各写一套互不相干的规则。
2. 玩家能够稳定理解自身 HP、能量、Buff 与行为状态之间的关系。
3. 玩家在大招霸体、受击硬直、普通移动等状态下，会观察到符合直觉的响应差异。
4. 玩家在高频输入下仍然觉得角色“跟手”，不会频繁怀疑输入被吞或状态切换不合理。
5. 系统应支持后续做出有观赏性的取消连段、空连、反打与处决演出。

### 场景验收

| 场景           | 预期表现                                                            |
| -------------- | ------------------------------------------------------------------- |
| 攻击敌人       | 敌人 HP 减少，并触发符合敌人类型的受击反馈                          |
| 攻击木箱       | 木箱 HP 归零后播放碎裂，无能量获取                                  |
| Buff 生效      | 属性实时变化，Buff 到期自动移除且属性复原                           |
| 玩家完成击杀/处决/弹反 | 玩家按已定义规则获得能量，不要求普通命中直接加能量              |
| HP 归零        | 触发死亡结果；具体玩家重生去向遵循独立 Respawn 规格                 |
| 大招期间被攻击 | Stats 正常扣血，但 Context 检测到 SuperArmor，**不进入硬直状态**    |
| 状态冲突       | 高优先级动作（受击）可打断低优先级（移动），反之不可                |

## 架构设计输入

- `CharacterContext` 是否需要拆分为更小的运行时协作对象，以避免职责膨胀。
- `可破坏道具` 是否只接入 `CharacterStats` / `IDamageable`，还是需要轻量行为层桥接。
- `NPC` 的 HP 支持应如何做成可选而非默认强绑定。
- 受击后的数值变化与状态切换应如何解耦，避免 `CharacterContext` 过度承担编排责任。
- 统一角色系统与 Respawn、Combat、Buff、Animation 等模块的边界如何稳定定义。

## 跨阶段预埋计划

| 阶段 | CharacterStats                            | CharacterContext                              |
| ---- | ----------------------------------------- | --------------------------------------------- |
| P0-A | 基础结构 + HP + `IDamageable`（陷阱扣血） | 基础结构 + 状态标志 + 组件引用 + 动画桥接     |
| P0-B | HitData → TakeDamage + 能量槽 + Buff 框架 | 受击决策协调（SuperArmor 判断）+ HitStun 状态 |
| P0-C | 处决 → 能量积攒 + 连招 Buff               | 动画取消窗口标志 + 连招状态切换协调           |
| P0-D | 能量消耗 → 大招释放                       | SuperArmor 激活 + Ultimate 状态管理           |

## 阶段完成定义

### P0-A 完成定义

- `CharacterStats` 与 `CharacterContext` 的基础职责边界已确认。
- Player 至少具备 HP、受击入口、基础状态标志、组件引用与动画桥接骨架。
- 统一接口已能覆盖 Player、Enemy、可破坏道具的最小闭环。
- 不要求在 P0-A 完成完整 Buff、能量循环、大招系统。

### P0-B 完成定义

- 受击后的数值结果与行为响应形成稳定闭环。
- 能量槽来源规则与 Buff 框架可被验证。
- 高优先级受击/霸体逻辑具备清晰结果。

### P0-C 完成定义

- 连招相关的状态切换协调与 Buff 联动建立完成。

### P0-D 完成定义

- 能量消耗、大招释放与霸体规则形成完整闭环。

## Review

### 已确认决策

- `CharacterStats` 是统一数值层，`CharacterContext` 是行为层运行时枢纽。
- `CharacterContext` 需要收敛职责边界，不应承担重生判定、房间规则或完整数值结算。
- `NPC` 的 HP 支持为可选，不要求所有 NPC 默认纳入完整受伤/死亡体系。
- Player 能量首版来源按“击杀 / 处决 / 弹反”定义，不写成“普通命中即获得能量”。
- 玩家死亡后的具体重生结果不在本规格内写死，遵循独立 Respawn 规格。

### 架构待细化项

- `CharacterContext` 是否需要进一步拆分，以及如何拆分。
- 受击响应协调应由哪些运行时模块共同承担。
- 可破坏道具与 NPC 在统一角色系统中的最小接入模型。
