# Feature Spec: Metroidvania Foundation

## Feature 简述

本项目是一款以**连通世界探索 + 动作战斗 + 能力门回访**为核心循环的 `Metroidvania-like` 游戏。

这份文档定义项目级设计骨架，用于统一后续的区域设计、能力解锁、战斗节奏、房间职责和内容投放逻辑。

## 一句话定位

一款强调**高质量动作手感**与**能力驱动探索回访**的 2D `Metroidvania-like` 动作平台游戏。

## 核心体验目标

- 玩家在探索中不断看到“现在过不去，但以后一定能回来打开”的路径承诺。
- 新能力不仅提升角色操作表达，也要让旧地图产生新的路线价值。
- 战斗、平台跳跃、解谜、捷径开启应共同服务于探索推进，而不是彼此割裂。
- 每个区域都应有明确的空间身份、挑战主题和奖励预期。

## 核心循环

`探索 -> 遭遇阻挡 -> 获得能力/钥匙/世界状态变化 -> 打开新路径 -> 回访旧区域 -> 发现捷径/隐藏物/新挑战`

## 设计支柱

### 1. 连通世界

- 世界结构优先采用多区域互联，而不是完全独立的关卡列表。
- 区域之间应通过主干连接、侧路、单向开启捷径、枢纽房间形成可记忆的导航结构。
- 玩家在地图认知上要能回答三个问题：我在哪、我被什么挡住、我以后为什么还要回来。

### 2. 能力门驱动推进

- 主线推进主要由能力门、关键道具门、机关状态门组成。
- 每种能力门都要具备明显的视觉提示，避免玩家遗忘或误判。
- 新能力必须至少满足以下之一：开启新区域、缩短回程、提升战斗/平台跳跃表达、解锁隐藏奖励。

### 3. 房间职责清晰

- 每个房间至少有一个主职责： traversal、combat、secret、upgrade、checkpoint、narrative、connector。
- 房间目标应单纯明确，不要在早期设计中把过多机制硬塞进同一空间。
- 连接房与奖励房需要节奏差异，避免整张地图持续高压。

### 4. 战斗服务探索节奏

- 战斗是探索过程中的强对抗节点，而不是把世界切碎成独立竞技场。
- 小怪密度、危险配置、回复资源都要考虑回访成本与存档点距离。
- Boss、精英怪、机关房应承担“进度门”与“能力检验”的双重作用。

## 世界结构骨架

### 推荐世界层次

- `Hub / Safe Zone`：功能汇聚、叙事回收、分流到多个区域。
- `Primary Region`：主线区域，承载阶段目标与能力门教学。
- `Sub Region`：围绕主区域扩展的可选支线或难度变体区域。
- `Challenge Room`：高风险高回报的短内容节点。
- `Secret Route`：依赖观察、技巧或后期能力的隐藏通路。

### 房间类型建议

| 类型 | 作用 | 典型内容 |
| --- | --- | --- |
| Connector | 串联路线，建立空间认知 | 电梯井、长桥、过道、竖井 |
| Traversal | 教学或检验移动能力 | 连续跳台、墙跳段、空中换向 |
| Combat | 房间级战斗压力 | 小怪组合、精英守门 |
| Upgrade | 给予持久奖励 | 新能力、血量扩容、资源上限 |
| Checkpoint | 节奏缓冲与回收 | 存档点、恢复点、传送锚点 |
| Secret | 强化探索回报 | 隐藏墙、假地板、迂回路径 |
| Narrative | 提供世界信息 | NPC、壁画、遗迹、事件房 |

## 进度门类型

### Traversal Gates

- 二段跳
- 冲刺穿越
- 墙爬 / 壁跳强化
- 钩索 / 弹射点
- 缩体 / 形态切换
- 水下移动 / 抗性能力

### System Gates

- 钥匙或印记
- 开关或机关联动
- 击败 Boss 后的世界状态变化
- 特定战斗技能检查
- NPC 事件推进

## 奖励投放原则

- 主线路径奖励优先给推进世界所需的关键能力或关键道具。
- 支线路径奖励优先给强化探索效率、战斗表达或资源上限的内容。
- 隐藏房奖励要明显高于到达成本，否则会削弱探索成就感。
- 奖励要尽量改变玩家决策，而不只是数值微调。

## 区域设计检查表

- 该区域的空间主题是否一句话可描述。
- 该区域是否至少引入一个可记忆的阻挡点。
- 该区域是否提供至少一个未来回访理由。
- 该区域是否有清晰的 checkpoint 与捷径策略。
- 该区域的敌人、地形、机关是否服务同一个主题。

## 能力设计检查表

- 该能力是否同时影响移动、战斗、探索三者中的至少两者之一。
- 该能力是否能在旧区域中创造至少两个新的可达点。
- 该能力是否有可读的世界提示，帮助玩家主动回想被阻挡路径。
- 该能力是否需要同步接入地图标记、存档数据、教程提示和相机 framing。

## 里程碑建议

### P0：灰盒验证

- 完成基础移动、战斗、受击、存档点原型。
- 建立 1 个 Hub + 2 个小区域 + 2 种能力门。
- 验证“发现阻挡 -> 解锁能力 -> 回访打开捷径”的最小闭环。

### P1：主干成型

- 扩展到可持续游玩的区域骨架。
- 建立稳定的升级、地图、存档、捷径系统。
- 让每个区域具备独立视觉主题与敌人主题。

### P2：内容深化

- 扩展支线、隐藏房、挑战房、可选 Boss。
- 提升区域互联度和路线选择深度。
- 加强叙事回收、世界状态变化和高阶能力联动。

## 对后续文档的约束

- `CharacterFeature`、`AbyssRespawn`、地图系统、能力系统等后续文档都应遵守本骨架。
- `docs/ModuleDesign/ProjectArchitecture.md` 应作为项目模块依赖与开发顺序的顶层参考。
- 若新增能力、区域或系统不能强化“探索 -> 解锁 -> 回访”循环，应重新评估优先级。
- 若某项设计更像纯关卡制动作游戏机制，需明确说明其如何适配 `Metroidvania-like` 世界结构。
